# Support Page Spec

> **Status:** Ready for Implementation
> **Created:** 2026-03-20
> **Author:** User + Codex Interview

## Problem Statement
Visitors need a simple public support page on the existing static site so they can contact Elevenstoic by email.

## Success Criteria
- [ ] Navigating to `/support/` loads a public support page on the static site.
- [ ] The page shows only a heading and a clickable `mailto:` link for `hello@elevenstoic.com`.
- [ ] The page uses the existing Elevenstoic visual language without adding forms, auth, or extra support flows.

## Scope

### In Scope (v1)
- Add a new public support page under `/support/`.
- Reuse the site's existing visual style.
- Link directly to `mailto:hello@elevenstoic.com`.

### Out of Scope
- Homepage navigation changes.
- Contact forms, FAQs, chat, or additional support methods.
- Copy-to-clipboard or other fallback UI.
- Access control or role-based behavior.

## User Stories

### Visitor
As a visitor, I want a simple support page with an email link so that I can contact Elevenstoic directly.

**Acceptance:**
- Given I open `/support/`, when the page loads, then I see a support heading and the email link.
- Given I click the email link, when my device supports email handlers, then it opens a compose flow to `hello@elevenstoic.com`.

## Technical Constraints
- Must work as a static HTML page in the current repository structure.
- Must not affect the existing landing page behavior.

## Test Strategy
- Public interface(s) to test: static route `/support/` and the rendered email link.
- Unit test focus: none.
- Integration test focus: manual browser verification of static route and `mailto:` target.
- Required invariant tests:
  - [ ] None.

## Edge Cases
| Scenario | Expected Behavior |
|----------|------------------|
| No mail client configured | The page still shows the email address as visible text and link text. |
| Direct navigation to `/support/` | The static support page loads independently of the landing page. |
